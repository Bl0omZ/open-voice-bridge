# 添加新设备适配器

新设备必须从事实和失败路径开始，而不是复制 Xiaomi 类名后替换产品名称。

## 1. 建立研究 profile

在 `device-profiles/` 新建设备 JSON，按 `specs/device-profile.schema.json` 填写：

- 精确产品与型号；
- 可证明的设备身份；
- 每条候选传输及其角色；
- 能力；
- 各平台状态；
- 官方文档、公开协议或可复核的参考来源。

初始状态通常是 `research`。只有代码、自动验证和目标平台真机门都完成后，相关平台才能改成 `implemented`。

## 2. 先判断电脑看到的是什么

按以下顺序采证：

1. 系统音频设备：是否已经作为麦克风或输出出现；
2. USB：接收器或设备是否通过 USB 数字音频、串口或厂商接口出现；
3. Bluetooth HID：是否有按键 report；
4. BLE GATT：service、characteristic、通知和控制写入；
5. BR/EDR 或其他厂商无线链路：只有存在平台可用入口时才实现。

“产品支持蓝牙”不能推导出“macOS/Windows/Linux 能把它当麦克风”。

## 3. 定义适配器输出

设备适配器只向上层输出通用事件：

- voice stream started/stopped；
- 带格式的音频帧；
- logical button down/up；
- battery/device status；
- adapter error / disconnected。

平台按键、系统快捷键和虚拟音频选择留给 platform backend。

## 4. 建立失败关闭门

至少覆盖：

- 名称相似但不是目标设备；
- 传输存在但协议或音频格式不符合；
- 初始化超时、订阅未生效、迟到回调；
- 权限在运行中撤销；
- 设备断开时仍有按键按下或音频流；
- 两台同型号设备或多路发射器的身份隔离。

## 5. 平台验收

每个平台独立完成：

- 安装与权限；
- 连接、重连和退出清理；
- 音频格式、延迟和持续输入；
- 按下/松开与重复事件；
- 不修改默认音频、不保存语音；
- 打包、签名或未签名边界说明。

一个平台的 PASS 不自动转移到另一个平台。
